# Sponsor Sophia — the Wisdom Gate

Sophia is and always will be **Apache-2.0-licensed and free**. Sponsorship funds the *time and
compute* to keep it honest: development, benchmark runs, and — most importantly — the
**independent third-party validation** the failure ledger says is still missing.

> **Wisdom before intelligence.** You are funding rigorous, fail-closed source discipline
> with a public ledger of what is **not** yet proven — not an AGI claim.

## What your sponsorship funds

- Maintainer time on the open core (gate, corpus, verifiers, MCP).
- Compute for reproducible benchmark runs and the live-grounding backend.
- The honest-credibility priorities from the [failure ledger](agi-proof/failure-ledger.md):
  a **third-party-authored** eval pack and **human-rater** review.

**Important integrity rule（重要誠信規則）**: Sponsorship buys **recognition and (at higher tiers) input on direction**. It never buys control over what counts as true, benchmark results, or any AGI claim. See the full rules below.

## What sponsorship does **NOT** buy (on purpose)

To protect the one thing that makes this project worth funding — its integrity — sponsorship
**cannot**:

- **Buy editorial control over what is true.** Sponsors may influence *direction* (which
  domain to work on next), never *content or verdicts*. The gate stays neutral.
- **Put your name in the training data.** Recognition lives in this file and release notes,
  never inside `training/` examples (that would corrupt the provenance corpus).
- **Buy a benchmark result, a "validated" label, or any AGI claim.** Numbers still have to
  clear the no-overclaim gate; sponsorship changes nothing about the evidence bar.
- **Buy guaranteed turnaround or support SLAs** (that's a paid [services](docs/07-Growth/SERVICES.md)
  engagement, not sponsorship).

## Maintainer notes: Patreon sync

1. Set up your custom tiers on Patreon first (full guide + copy-paste text: [docs/07-Growth/Patreon-Tier-Setup.md](docs/07-Growth/Patreon-Tier-Setup.md)). Use the exact titles: `Wisdom Seed (智慧種子)`, `Gatekeeper (守門人)`, etc.
2. Copy the **Creator Access Token** (and optionally refresh + client id/secret) into `.env` (never committed).
3. Test locally:

   ```bash
   python tools/sync_patreon_supporters.py                 # preview
   python tools/sync_patreon_supporters.py --update        # edits SPONSORS.md
   python tools/sync_patreon_supporters.py --write-json    # updates data/patreon_supporters.json
   ```

4. For GitHub Actions: add these **repository secrets** (Settings → Secrets and variables → Actions):
   - `PATREON_CREATOR_ACCESS_TOKEN`
   - (optional) `PATREON_CREATOR_REFRESH_TOKEN`, `PATREON_CAMPAIGN_ID`, `PATREON_CLIENT_ID`, `PATREON_CLIENT_SECRET`

5. Trigger the workflow manually from the Actions tab ("Sync Patreon Supporters") or let the weekly schedule run.

**Posting supporter updates back to Patreon**  
After syncing, you can easily create nice thank-you posts for your patrons using:
```bash
python tools/post_to_patreon.py --supporter-post --dry-run
```
See `docs/07-Growth/Patreon-Tier-Setup.md` → "Helper mode for huge supporter expansions".
6. The sync only includes **active_patron** status + non-empty public `full_name`.
7. `tools/build_web_data.py` will pick up a count + tier summary for the public manifest when you next rebuild the site data.
